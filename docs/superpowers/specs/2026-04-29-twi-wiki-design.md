# TWI SIG Wiki — Design

**Date:** 2026-04-29
**Goal:** Build a local, browsable wiki of the Confidential Computing Trustworthy Workload Identity (TWI) SIG mailing list, by scraping the public groups.io archive and feeding the threads into [llmwiki](https://github.com/lucasastorian/llmwiki).

## Scope

- One-shot snapshot of the archive at the time the scraper is run. No incremental refresh, no scheduled job.
- Read-only ingest. No posting, no auth, no member-only sections (`/files`, etc.).
- Source: `https://lists.confidentialcomputing.io/g/Trustworthy-Workload-Identity-SIG`. As of 2026-04-29: 119 topics, 169 messages, public archive.

## Tools and Dependencies

- **Scraper:** Python 3.11+. Deps: `curl_cffi` (Chrome-impersonating HTTP client), `beautifulsoup4` + `lxml` (HTML parsing), `python-dotenv`. Managed via `pyproject.toml`.
- **Auth:** groups.io blocks unauthenticated traffic with a "Groups.io AI Crawler Policy" 402 page, identified by both User-Agent and TLS fingerprint (JA3/JA4). The scraper bypasses this by:
  1. Using `curl_cffi` with `impersonate="chrome"` to match a real Chrome TLS handshake.
  2. Sending the user's groups.io session cookie (loaded from `.env` as `GROUPSIO_COOKIE`).
  An earlier design used Firecrawl, but Firecrawl's chrome-cdp + tlsclient + stealth-proxy engines were all 402-blocked by groups.io, so it was dropped.
- **llmwiki:** vendored under `llmwiki/`. Requires Python 3.11+ and Node.js 20+ for its own setup (`pip install -r api/requirements.txt`, `npm install` in `web/`). This repo's Makefile delegates to llmwiki's CLI; we don't fork its code.

## Data Flow

1. **Discover topics.** Fetch `/g/Trustworthy-Workload-Identity-SIG/topics` (paginated via `?page=N`). Parse out topic IDs and URLs from anchor tags. Write the list to `data/raw/topics.json`.
2. **Fetch threads.** For each topic ID:
   - Fetch `/g/.../topic/<id>` to get the ordered list of message IDs in that thread.
   - For each message ID, fetch `/g/.../message/<id>` and parse out headers (sender, date, subject) and body HTML, converting body to markdown.
   - Cache each raw HTML response under `data/raw/topic-<id>.html` and `data/raw/message-<id>.html` so reruns skip the network.
3. **Render.** For each topic, write `data/threads/<id>-<slug>.md` with the format below. Slug is derived from the subject.
4. **Index with llmwiki.** `cd llmwiki && ./llmwiki init ../data/threads/` then `./llmwiki serve ../data/threads/`. The wiki is generated by Claude over MCP, per llmwiki's standard flow. Output lives under `llmwiki/wiki/` (gitignored).

### Thread Markdown Format

```markdown
---
topic_id: 42
subject: "..."
participants: ["alice@...", "bob@..."]
first_post: 2025-04-01
last_post: 2025-04-05
message_count: 3
source: https://lists.confidentialcomputing.io/g/Trustworthy-Workload-Identity-SIG/topic/42
---

# <Subject>

## Message 1 — Alice <alice@...> — 2025-04-01

<body>

## Message 2 — Bob <bob@...> — 2025-04-03

<body>
```

## Make Targets

- `make scrape` — runs `scripts/scrape.py`. Idempotent thanks to the raw cache.
- `make build` — first-time llmwiki init + index over `data/threads/`.
- `make serve` — runs llmwiki's API + web UI, browser at `http://localhost:3000`.
- `make clean-cache` — removes `data/raw/`. Does not touch `data/threads/`.

## Error Handling

- HTTP 4xx/5xx: log to `data/raw/failures.log` with the URL and status, skip the item, continue. Reruns retry only the missing items.
- Parsing failure on a page (unexpected DOM shape): log + skip + record in `failures.log`.
- 402 responses (cookie expired or AI-policy block): hard-fail with a clear message asking the user to refresh `GROUPSIO_COOKIE`.
- No exponential backoff; the cache makes reruns cheap. Add a small inter-request delay (`~0.5s`) to be polite.

## Out of Scope

- Incremental updates / RSS polling.
- Members-only `/files` section (not relevant for this wiki).
- Posting / write operations on groups.io.
- Forking or modifying llmwiki itself.

## Open Risks

- **Cookie expiry.** Groups.io session cookies will eventually expire. When that happens, the user re-pastes a fresh cookie into `.env` and reruns. We surface the failure clearly rather than retrying.
- **Topic-list pagination mechanism.** Groups.io paginates topics; the `?page=N` shape will be confirmed during implementation by inspecting one paginated response.
- **Message body HTML quality.** Groups.io email bodies vary (plain text, multipart, quoted replies). `render.py` strips reply quotes and converts to markdown using a simple HTML→MD pass; messy edge cases may need targeted fixes.
