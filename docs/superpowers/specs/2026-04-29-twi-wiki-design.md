# TWI SIG Wiki вЂ” Design

**Date:** 2026-04-29
**Goal:** Build a local, browsable wiki of the Confidential Computing Trustworthy Workload Identity (TWI) SIG mailing list, by scraping the public groups.io archive and feeding the threads into [llmwiki](https://github.com/lucasastorian/llmwiki).

## Scope

- One-shot snapshot of the archive at the time the scraper is run. No incremental refresh, no scheduled job.
- Read-only ingest. No posting, no auth, no member-only sections (`/files`, etc.).
- Source: `https://lists.confidentialcomputing.io/g/Trustworthy-Workload-Identity-SIG`. As of 2026-04-29: 119 topics, 169 messages, public archive.

## Tools and Dependencies

- **Scraper:** Python 3.11+. Deps: `httpx` (Firecrawl HTTP client), `python-dotenv`, `pydantic` (response models). Managed via `pyproject.toml`.
- **Firecrawl:** accessed through the user-provided proxy at `https://api.clawdi.ai/proxy/firecrawl` with a bearer token. Endpoint `/v1/scrape` returns markdown. API key loaded from `.env`.
- **llmwiki:** vendored under `llmwiki/`. Requires Python 3.11+ and Node.js 20+ for its own setup (`pip install -r api/requirements.txt`, `npm install` in `web/`). This repo's Makefile delegates to llmwiki's CLI; we don't fork its code.

## Data Flow

1. **Discover topics.** Crawl `/g/Trustworthy-Workload-Identity-SIG/topics` (paginated). Parse out topic IDs and URLs. Write the list to `data/raw/topics.json`.
2. **Fetch threads.** For each topic ID:
   - Scrape `/g/.../topic/<id>` to get the ordered list of message IDs in that thread.
   - For each message ID, scrape `/g/.../message/<id>` and capture the markdown body plus headers (sender, date, subject).
   - Cache each Firecrawl response under `data/raw/message-<id>.json` and `data/raw/topic-<id>.json` so reruns skip the network.
3. **Render.** For each topic, write `data/threads/<id>-<slug>.md` with the format below. Slug is derived from the subject.
4. **Index with llmwiki.** `cd llmwiki && ./llmwiki init ../data/threads/` then `./llmwiki serve ../data/threads/`. The wiki is generated by Claude over MCP, per llmwiki's standard flow. Output lives under `llmwiki/wiki/` (gitignored).

## Make Targets

- `make scrape` вЂ” runs `scripts/scrape.py`. Idempotent thanks to the raw cache.
- `make build` вЂ” first-time llmwiki init + index over `data/threads/`.
- `make serve` вЂ” runs llmwiki's API + web UI, browser at `http://localhost:3000`.
- `make clean-cache` вЂ” removes `data/raw/`. Does not touch `data/threads/`.

## Error Handling

- Firecrawl 4xx/5xx: log to `data/raw/failures.log` with the URL and status, skip the item, continue. Reruns retry only the missing items.
- Parsing failure on a page (unexpected DOM shape): log + skip + record in `failures.log`.
- No exponential backoff; the cache makes reruns cheap.

## Out of Scope

- Incremental updates / RSS polling.
- Members-only `/files` and `/wiki` sections.
- Anything that requires posting or authenticated access.
- Forking or modifying llmwiki itself.

## Open Risks

- **Firecrawl markdown quality on groups.io.** If message bodies render poorly, `render.py` may need an HTML-fallback path. Will be evaluated during implementation against the first few messages.
- **Topic-list pagination mechanism.** Groups.io supports `?start=` query offsets; the exact form will be confirmed during implementation by inspecting one paginated response.
